String 类的常用方法都有那些

length charAt substring concat indexOf replace trim

抽象类中是否一定要有抽象方法

不一定 但有抽象方法的类必须声明为抽象类

抽象类能使用 final 修饰吗？

不能，抽象类是被用于继承的，final修饰代表不可修改、不可继承的。

BIO、NIO、AIO 有什么区别

BIO是同步阻塞 一个连接一个线程，用于连接少 NIO是同步非阻塞 一个请求一个线程，用于连接多短连接 AIO是异步非阻塞 一个有效请求一个线程，用于连接多长连接（即下次访问依然保持TCP连接不会关闭）

并发与并行

并发是同一时间段各事件公用一个cpu运行 并行是同一时间各事件使用不同cpu同时进行

守护线程

用于服务于其他线程，如垃圾回收线程 只有jvm停止才会退出

说一下 runnable 和 callable 有什么区别

前者返回值为void 自己处理异常 不能上抛
后者有返回值，异常可上抛

notify和notifyall

前者随机唤醒一个wait 后者唤醒所有wait 看优先级排序

线程的 run()和 start()有什么区别

run只是一个普通的方法 只有start启动线程 让THread去调用run方法

线程池中 submit()和 execute()方法有什么区别

submit包含execute 只是比execute多了返回值这一段

锁的升级

synchronized是重量级锁，耗费资源
jvm优化出 偏向锁 轻量级锁 和 重量级锁
偏向锁是直接保存对象的ID到线程锁对象的对象头，每次值比较ID即可 在1个线程时使用
轻量级锁是存在2个线程 另一个线程通过自旋获取锁 在多个线程但访问错开使用
如果2个线程访问有冲突 升级为重量级锁

如何避免死锁

按同一顺序访问对象 不会存在需要的资源被后面的对象调用从而发生互相等待的死锁
避免事务中的用户交互 防止用户没及时反馈造成
保持事务简短并在一个批处理中 事务越短 等待的对象越少
使用低隔离级别 允许对象读取未完成的对象的数据

ThreadLocal 是什么？有哪些使用场景

线程共享变量 原理也是map键值对
用于做参数传递
在spring中用于管理数据库的连接
数据库连接时避免A连接关闭了B连接

synchronized原理

每个对象由三部分组成：对象头 实例变量和填充数据
对象头中包含由monitor 当线程访问时会进入EntryLIst等待执行monitorenter获取monitor持有权以进入owner区 获取锁后进入owner中运行monitor计数器加1其他线程不能进入 退出时执行monitirexit计数器减1 如果调用wait方法会进入waitset中
而synchronized中只有一个常量供jvm判断是不是同步代码，是的话jvm才调用同步方法，不是则不调用

volatile和synchronized的区别

volatile本质是在告诉jvm当前变量在寄存器（工作内存）中的值是不确定的，需要从主存中读取； synchronized则是锁定当前变量，只有当前线程可以访问该变量，其他线程被阻塞住。
volatile仅能使用在变量级别；synchronized则可以使用在变量、方法、和类级别的
volatile仅能实现变量的修改可见性，不能保证原子性；而synchronized则可以保证变量的修改可见性和原子性
volatile不会造成线程的阻塞；synchronized可能会造成线程的阻塞。
volatile标记的变量不会被编译器优化；synchronized标记的变量可以被编译器优化

Lock与synchronized 的区别

lock等待锁一段时间会停止等待 去干别的事情而synchronized会一直等
lock要手动释放 synchronized不用
lock一直等待 trylock获取不到立即返回 或者等待一段时间返回
在资源竞争不是很激烈的情况下，偶尔会有同步的情形下，synchronized是很合适的。

序列化问题

为什么序列化对象通常要重写 hashCode 和 equals方法

2个内容完全一样的对象，在我们“人”看来就是一样的对象，但是计算机不这么认为，为了保证对象在进行比较时候符合人们的思维习惯，所以我们需要重写这2个方法。
在向 set 这种集合中添加对象的时候，set 要求对象都是不重复的，这个检查重复的过程，首先是检查对象的 hashCode() 值，如果这个值一致的话再去通过 equals 方法进行比较，如果 equals方法返回true，那么这个对象就不会被加入到 set 当中
对象的序列化主要有两种用途：
把对象的字节序列永久地保存到硬盘上，通常存放在一个文件中；
在网络上传送对象的字节序列。
在很多应用中，需要对某些对象进行序列化，让它们离开内存空间，入住物理硬盘，以便长期保存。比如最常见的是Web服务器中的Session对象，当有 10万用户并发访问，就有可能出现10万个Session对象，内存可能吃不消，于是Web容器就会把一些seesion先序列化到硬盘中，等要用了，再把保存在硬盘中的对象还原到内存中。
当两个进程在进行远程通信时，彼此可以发送各种类型的数据。无论是何种类型的数据，都会以二进制序列的形式在网络上传送。发送方需要把这个Java对象转换为字节序列，才能在网络上传送；接收方则需要把字节序列再恢复为Java对象

为什么要使用SerialversionUID

如果用户没有自己声明一个serialVersionUID,接口会默认生成一个serialVersionUID
但是，如果我们不自己声明时，
当我们修改Person类的时候，Person类对应的SerialversionUID也变化了，而序列化和反序列化就是通过对比其SerialversionUID来进行的，一旦SerialversionUID不匹配，反序列化就无法成功。在实际的生产环境中，我们可能会建一系列的中间Object来反序列化我们的pojo，为了解决这个问题，我们就需要在实体类中自定义SerialversionUID
不管我们序列化之后如何更改我们的Person（不删除原有字段），最终都可以反序列化成功

对于不同情况，即需要排序的数据的大小n
当n较小时直接使用选择或插入排序
若数据大部分有序的可以选择冒泡或快速排序
但是，要判断是否基本有序的标准很难拿捏，所以…

switch 是否能作用在byte 上，是否能作用在long 上，是否能作用在String上？

答：在Java 5以前，switch(expr)中，expr只能是byte、short、char、int。从Java 5开始，Java中引入了枚举类型，expr也可以是enum类型，从Java 7开始，expr还可以是字符串（String），但是长整型（long）在目前所有的版本中都是不可以的。
